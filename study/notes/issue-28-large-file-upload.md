# Note — 대용량 파일 업로드 (청크 / S3 멀티파트)

> 관련 이슈: #28 · 관련 구현: #10(단일 multipart) · 성격: 개념/설계 노트(대화 정리)

## 한 줄 요약
작은 파일은 한 방에(단일 multipart), 큰 파일은 **프론트가 잘라서** 여러 번(청크) 또는 **S3로 직행**(presigned). 서버는 되도록 파일 데이터의 "통로"가 되지 않게 한다.

## 0. 먼저: getBytes(통째) vs getInputStream(스트림)
업로드 파일 내용을 다루는 두 방식. 차이는 **메모리(RAM)**.
```java
byte[] all = file.getBytes();               // 통째로: 파일 크기만큼 RAM 점유 (100MB → 100MB)
InputStream in = file.getInputStream();     // 스트림: 조금씩(버퍼 몇 KB)만 거쳐감
Files.copy(file.getInputStream(), target);  // 흐르는 대로 디스크에 바로 흘려담기
```
| | getBytes(통째) | getInputStream(스트림) |
|---|---|---|
| 메모리 | 파일 크기만큼 | 몇 KB(고정) |
| 대용량 | 위험/불가(OOM) | 안전 |
| 동시 업로드 | 부하 큼(RAM×인원) | 강함 |
| 용도 | 작은 파일 통째 가공(썸네일/해시) | 디스크 저장/전달 |
- 비유: 통째=큰 통에 물 100L 다 받고 옮김 / 스트림=수도관 열어 흐르는 대로 흘려보냄.
- #10 우리 코드는 `Files.copy(getInputStream())`라 파일을 통째로 메모리에 안 올림(대용량 첫 함정 회피).

## 1. 단일 요청 multipart (#10, 현재)
- 파일 하나를 HTTP 요청 1번에 통째로. `@RequestParam MultipartFile`.
- 스트리밍 저장이라 메모리는 안전하지만:
  - 끊기면 **처음부터** 다시(이어올리기 X), 프록시/서버 **타임아웃**, 진행률 뭉툭.
- 결론: ~수십MB엔 충분. 우리 학습 게시판(10MB 제한)에 적합.

## 2. 청크 업로드 — "프론트가 자른다"
- 대용량은 **프론트가 파일을 조각(chunk)으로 slice** 해서 여러 요청으로 보냄(`Blob.slice()`).
- 서버는 chunk를 임시로 모았다가 **다 오면 merge** + 메타 기록.
```
[프론트]  5MB씩 slice → chunk1, chunk2, ... (순차/병렬)
[서버]    각 chunk 접수 → 오프셋 관리 → 완료 시 하나로 합침
```
- 핵심: **자르는 건 프론트, 받아 합치는 건 서버**. (서버가 원본을 자르는 게 아님 — 이미 다 도착해야 자를 수 있으니 의미 없음)
- 직접 만들기보단 표준/라이브러리: **tus 프로토콜**, 프론트 **Uppy / resumable.js** → 이어올리기·진행률·병렬을 처리.

## 3. S3 멀티파트 + presigned URL (권장, 프로덕션)
```
1) 프론트: "이 파일 올릴래" → 백엔드에 요청
2) 백엔드: S3 멀티파트 시작 + 각 파트 presigned URL 발급 → 프론트에 전달
3) 프론트: 파일을 파트로 잘라 그 URL로 S3에 "직접" 업로드 (백엔드 우회!)
4) 완료: 백엔드가 S3 complete 호출 + DB에 메타 기록
```
- **핵심 이점: 파일 바이트가 앱 서버를 안 지나감** → 프론트 → S3 직행.
  - 앱 서버 대역폭/디스크 부담 0(메타만 처리), S3가 내구성·확장·resume 담당, presigned라 크레덴셜 노출 없음.

## 규모별 선택
| 규모 | 방법 | 이유 |
|---|---|---|
| 작은 파일(~수십MB) | 단일 multipart(#10) | 단순·충분 |
| 중대형 / 불안정 네트워크 | 청크(tus/Uppy) | 이어올리기·진행률 |
| 대용량 / 프로덕션 | S3 멀티파트 + presigned | 서버 우회·확장성 최고 |

## 우리 프로젝트에는
- 학습 게시판 + 10MB 제한 → **단일 multipart가 딱 맞음**. 청크/S3는 오버스펙.
- 그래도 실무 감각으로 기억: "서버를 파일 통로로 만들지 말기 + 스트리밍/청크".
