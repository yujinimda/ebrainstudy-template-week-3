# Issue 01 — 3주차 오리엔테이션: REST API란 + 서버/프론트 분리

> 관련 PR: (오리엔테이션 — 코드 없음) · 참고: 2주차 게시판(Thymeleaf 버전)

## 무엇을 만드나
코드를 짜기 전, 3주차 전체 그림을 잡는 단계. "2주차 게시판을 REST API 서버 + Next 프론트로 쪼개면 뭐가 달라지나"를 이해한다.

## 알아야 할 개념
- **REST API = 화면이 아니라 데이터(JSON)를 준다.**
  - 2주차: 서버(Thymeleaf)가 `<table>...</table>` HTML을 통째로 그려서 응답.
  - 3주차: 서버는 `{ "content": [ ... ] }` 같은 JSON만 응답. 화면 그리는 건 프론트(Next) 담당.
- **자원(URL) + 행위(HTTP 메서드).**
  - "무엇을"은 URL로, "어떻게"는 메서드로 나눠 표현.
  - 같은 URL `/api/boards/1` 에 `GET`이면 조회, `DELETE`면 삭제.

  | 하고 싶은 것 | REST |
  |---|---|
  | 목록 | `GET /api/boards` |
  | 상세 | `GET /api/boards/1` |
  | 등록 | `POST /api/boards` |
  | 수정 | `PUT /api/boards/1` |
  | 삭제 | `DELETE /api/boards/1` |

## 2주차와 뭐가 다른가
- 2주차는 URL에 행위가 들어갔다: `/board/write`, `/board/delete`.
- REST는 URL에서 행위를 빼고 **메서드로** 옮긴다. URL은 오직 자원만 가리킨다.
- 서버가 화면을 안 그리므로, 화면 상태·렌더링은 전부 프론트 책임이 된다.

## 고민해봐야 할 것
- 서버(`:8080`)와 프론트(`:3000`)가 **다른 출처(Origin)** 라서 새로 생기는 문제들:
  - **CORS**: 브라우저가 다른 출처 호출을 기본 차단 → 서버가 허용 헤더를 내려줘야 함 (#11)
  - **응답 규격 합의**: 성공/실패를 항상 같은 JSON 모양으로 감싸야 프론트가 편함 (#3)
  - **nginx 통합**: 한 도메인으로 묶으면 애초에 "다른 출처"가 아니게 되어 CORS가 사라짐 (#16)
- 진행 원칙: **API를 먼저 전부 완성(#2~#11) → 프론트(#12~#15) → nginx 통합(#16).**

## 막히면 볼 것
- Origin = 프로토콜 + 도메인 + **포트**. 3000과 8080은 포트가 달라 "다른 출처".
- DB 기동: `cd docker && docker compose up -d` (localhost:3310, 시드 board 24행)
